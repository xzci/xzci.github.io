我的项目设计

发表于 2018-02-01 | 分类于 项目

经历了几个项目后，终于有机会自己从头去设计并实现一个项目，虽然对接的人员有着初步的设计，但是我还是根据现有的流程进行了改进。
并完成了coding， test， document等一系列事件。

理解式样

对于coding，如有式样，那么应该先预留时间去理解式样，把坑，或者不理解的地方在理解式样的时候尽可能的找出来，做好前期的沟通。
自己根据自己的能力先对所需要花费的时间，进行预计。明确好自己应该要做什么。
如果式样有不明确的地方，可以自己再从新写一遍，让参与人员更好的去理解相关的参数和设置。

Coding

错误处理

针对可能出现的错误进行预期，如果可以通过相关的if检测出现，则不应该用异常来对程序做状态的修改，同时，要对整个流程熟悉，了解如果出错后，应该会什么方式去修正这个异常（程序最好不会因为异常而突然中止）。（修正异常也有可能是重新运行，良好的提示信息等）
当然，如何去获取这个异常，异常怎么处理，要首先的设计好，出现异常的流程是否会对接下来的流程产生问题。（<-需要提前做好沟通，并确认解决方式）

程序可读性

为了程序的可读性，一定要言而有信（信息），英文不好，用拼音，一定要浅显易懂，（不浅显易懂也可以，但是不能错），注释同理，其实在变量名规则等较为宽松，但是注释，函数名等重要信息一定不能有误，比如，保存[误写]为[编辑]等。这回在程序进行修改的时候，造成巨大的麻烦。

可能需要修改的地方

比如在程序生成文件时，这些文件在一开始的时候尽量不要 hard coding，在需要修改的时候，往往会出现漏改，错改的情况，一定在前期进行设计，统一变量，参数传递。改其一，则至全。
同时，也可以设计相应的config文件，程序运行时，先去读取config文件中的内容，再对程序进行交互。这些内容可以生成的路径，文件命名，密码等数据。依照的原理依然同上。
或者如果修改时间非常迅速，经常需要修改，可以将config文件更改为更加友好的交互界面。由用户或者二级程序员交互。

程序的原子化

首先更加简单的程序会对debug，测试带来便利，比如你的函数只实现了一个功能，那么你的单元测试就可以很快的实现，不需要考虑实现几个模块的相互依赖关系。
并且大大提升了可读性。

程序要经常备份

使用svn或者git对程序进行备份，svn相较git更为简单，（git我也不怎么熟悉）代码备份有很多的好处，首先防止误操作，可以将代码恢复至上一个commit的版本，并且，在使用某些软件进行编程的时候，如果程序出现崩溃，可以直接从服务器中拖去一份旧版在重新编辑。
如果不使用svn或其他版本管理器，那么如果崩溃，你所有的文件，代码丢失后，所造成的损失是无法估量的。
一定要备份

制作文档

制作一个清晰的文档是重中之重，我的看法是可以一边coding，一边制作文档，便于自己理清思路，（并且往往coding结束，就懒的制作文档了）文档可以给后期维护人员带来极大的便利（如果不要求文档，希望自己可以在余力中整理）（理解业务流程）

展开

我觉得还是跟同事所说的一样，

  • Domain knowledge
  • Data
  • Coding

按重要性区分。所以一定要理解业务。不能死编程。
